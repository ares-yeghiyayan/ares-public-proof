# Public Proof Discipline

This note is for readers who want to understand how I use GitHub as a public proof surface.

The point is not to show everything. The point is to show a disciplined way of working while keeping private work private.

## The signal

A useful public proof surface should show:

- small work units
- clear boundaries
- review before proof
- quality before claim
- repeatable progress
- clean documentation habits
- visible issue and pull request rhythm

## What I want a reviewer to see

I want a reviewer to see that I do not treat activity as proof by itself.

A commit is not enough.
A note is not enough.
A claim is not enough.

The work should be small enough to inspect, clear enough to understand, and bounded enough to avoid exposing sensitive material.

## What stays private

Private work stays private.

This public surface should not reveal internal architecture, private strategy, internal workflows, private prompts, customer assumptions, market strategy, sensitive implementation details, financial advice claims, performance claims, public release claims, or production readiness claims.

## Operating rule

```text
small work -> review discipline -> quality before claim -> safe boundary -> repeatable proof
```

## Why this matters

A serious public proof trail should build trust without leaking private work.

That is the balance this repository is designed to practice.
