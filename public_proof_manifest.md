# Public Proof Manifest

This manifest defines what can move from a private research space into a future public proof surface. It exists to prevent accidental exposure while still supporting a credible public GitHub profile.

## Safe to Publish

- Sanitized positioning
- High-level learning summaries
- Public-safe build rhythm notes
- Non-sensitive documentation examples
- Public issue and PR discipline notes
- CI badge and basic test-status notes
- Public learning log

## Needs Redaction

- Drafts that mention private methods or naming systems
- Notes that include internal assumptions
- Examples that include sensitive context
- Reports that include private planning language
- Notes that contain internal operating instructions
- Screenshots that show private repo names, branches, issues, account context, or private repository state

## Never Publish

- Private access material
- Private prompts
- Live operating details
- Customer data
- Private strategy
- Product architecture
- Module names
- Risk engine mechanics
- Decision workflow mechanics
- Target customer strategy
- Market positioning strategy
- Internal proof chain structure
- Private repository screenshots
- Naming of private internal components
- Proprietary edge logic
- Investor-private materials
- Personal documents

## Public Repo Minimum Viable Contents

- Public-safe README
- Public proof manifest
- Safe public scope document
- Redaction checklist
- One public-safe learning note
- One public-safe build rhythm note
- One public issue/PR discipline note
- One lightweight CI or markdown quality check

## Weekly Public Proof Rhythm

- Publish one small public-safe improvement.
- Open one issue before building.
- Use one pull request for each public proof step.
- Keep private research private until explicitly sanitized.
- Review the public surface weekly for accidental claims, exposure, or hype.
